# 01 · 결정 로그 (Decisions / ADR)

> 이미 내린 결정들. **다시 논쟁하지 말 것.** 뒤집을 땐 맨 아래에 새 항목으로 추가하고 이유를 적는다.
> 각 항목: 결정 / 이유 / 기각한 대안.

---

### D1. 네이티브 앱이 아니라 PWA로 간다
- **결정:** 모바일 퍼스트 웹을 PWA(매니페스트 + 서비스 워커)로 만들어 안드로이드 홈 화면에 설치.
- **이유:** 기능 대부분(히스토리·차트·보고서)이 "앉아서 보는" 읽기 작업이라 웹이 강함. "계속 기능을 붙인다"는 특성상, 배포 후 새로고침으로 끝나는 웹이 매번 빌드·재설치하는 네이티브보다 압도적으로 빠르다. 판매가 아니라 스토어 발견성·심사도 불필요. "앱처럼 보이는 것"(홈 아이콘·전체화면·카메라·오프라인·푸시)은 PWA가 다 제공.
- **기각:** 네이티브(빌드/재설치 반복, 본인용엔 과함), Fud AI 등 기존 오픈소스 포크(iOS+안드로이드 네이티브 2종 코드베이스라 웹 전환 시 재사용률 0).

### D2. 안드로이드만 대응
- **결정:** iOS 미지원.
- **이유:** 본인이 안드로이드 사용. iOS 웹푸시는 홈 화면 설치 상태에서만 동작하는 등 제약이 많은데, 그걸 신경 쓸 이유가 없음.
- **기각:** 크로스 플랫폼 대응(불필요한 복잡도).

### D3. 데이터는 Neon(Postgres)에 저장. 로컬 저장 안 씀
- **결정:** 모든 기록을 Neon Postgres에.
- **이유:** 브라우저 로컬 저장(IndexedDB)은 그 브라우저 안에 갇히고 캐시 삭제 시 소실됨. 폰·PC 양쪽에서 같은 데이터를 보려면 서버 DB가 필요. 이미 Neon에 익숙함.
- **기각:** IndexedDB/localStorage (동기화·백업 불가).

### D4. ORM은 drizzle-orm
- **결정:** drizzle-orm + Neon serverless driver.
- **이유:** 기존에 쓰던 스택. 타입 안전, 마이그레이션 깔끔, Vercel 서버리스 환경과 궁합 좋음.

### D5. 사진 분석은 Gemini 2.0 Flash
- **결정:** 멀티모달 LLM으로 Gemini 2.0 Flash 사용.
- **이유:** 음식 인식·칼로리 추정에 싸고 빠르고 충분. 구조화 JSON 출력 잘함. 이미 익숙.
- **참고:** "AI"는 별도 학습 모델이 아니라 그냥 사진+프롬프트를 LLM에 던지는 것. 모델 교체는 어댑터 한 군데만 바꾸면 되도록 추상화할 것 (`06` 참고).
- **기각:** 자체 비전 모델 학습(과함), CLIP+DB 매칭(MVP엔 불필요).

### D6. 배포는 Vercel
- **결정:** Vercel.
- **이유:** Next.js 궁합 최상, Vercel Cron으로 푸시 스케줄링까지 한 곳에서 해결.
- **참고:** Netlify도 Scheduled Functions로 가능은 하지만, Cron+서버리스 일관성 때문에 Vercel로 통일.

### D7. 인증은 "최소 보호" — 회원가입 없음
- **결정:** 단일 사용자용. 풀 인증 시스템 없이, 공개 URL이 그냥 노출되지 않도록 가벼운 보호만 건다.
- **이유:** 본인만 씀. 로그인 UX는 마찰만 추가.
- **방법:** `02-architecture.md`의 인증 섹션 참고 (단일 패스코드/쿠키 또는 환경변수 기반 미들웨어).

### D8. 차트는 Recharts
- **결정:** Recharts.
- **이유:** 기존 스택. 일/주/월 막대·라인 차트에 충분.

### D9. 푸시는 web-push + VAPID + Vercel Cron, 스마트 스킵
- **결정:** 표준 Web Push(VAPID 키) + `web-push` 라이브러리, 발송 스케줄은 Vercel Cron. 발송 전 "오늘 이미 기록했나" 확인해 기록 있으면 건너뜀.
- **이유:** 외부 푸시 서비스 없이 자체 완결. 잔소리 안 하는 알림.
- **순서:** 핵심 앱 완성 후 Phase 4에서 얹음 (곁가지 많아 처음부터 넣으면 디버깅 지점 증가).

### D10. 프로젝트명: 밥로그 (BabLog)
- **결정:** 한글 표기 "밥로그", 식별자/도메인 `bablog`.
- **이유:** 밥 + log. 발음 쉽고 뜻 명확. "오늘 밥로그 찍었어?"가 자연스러움.

---

### D11. AI 모델: Gemini 2.0 Flash → 3.1 Flash Lite로 갱신 (잠정)
- **결정:** D5의 모델 선택을 Gemini 3.1 Flash Lite로 갱신. 단, 착수 시 한 번 더 확인 예정인 잠정 결정.
- **이유:** D5 작성 시점 이후 더 최신 Flash 계열 모델이 나옴. 어차피 `lib/ai/` 어댑터 뒤에 격리되어 있어 교체 비용 낮음(D5 참고).
- **참고:** Phase 1 착수 시(실제 호출 코드 작성 시점) 모델명이 여전히 유효한지 재확인할 것.

### D12. 프로젝트 성격 재확인: 사내 프로젝트 아님
- **결정:** `04-tech-stack.md`에 있던 "기존 사내 프로젝트가 Next.js 16대" 언급은 다른 컨텍스트와 혼동된 것. 밥로그는 순수 개인 프로젝트이며 사내 프로젝트와의 호환성을 고려할 필요 없음.
- **이유:** 개발자 본인이 명확히 정정.

### D13. Next.js는 항상 착수 시점 최신 버전 사용
- **결정:** 특정 버전에 고정하지 않고 Phase 0 착수 시점의 최신 안정 버전을 그대로 사용. (2026-06-22 기준 16.2.9로 스캐폴딩.)
- **이유:** 개인 프로젝트라 레거시 호환 부담이 없음 (D12). Next.js 16에서 `middleware.ts`가 `proxy.ts`로 이름이 바뀌는 등 breaking change가 있었음 — 이후 세션도 작업 전 `node_modules/next/dist/docs/`로 현재 버전 컨벤션을 확인할 것 (`AGENTS.md` 참고).
