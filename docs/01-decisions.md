# 01 · 결정 로그 (Decisions / ADR)

> 이미 내린 결정들. **다시 논쟁하지 말 것.** 뒤집을 땐 맨 아래에 새 항목으로 추가하고 이유를 적는다.
> 각 항목: 결정 / 이유 / 기각한 대안.

---

### D1. 네이티브 앱이 아니라 PWA로 간다
- **결정:** 모바일 퍼스트 웹을 PWA(매니페스트 + 서비스 워커)로 만들어 안드로이드 홈 화면에 설치.
- **이유:** 기능 대부분(히스토리·차트·보고서)이 "앉아서 보는" 읽기 작업이라 웹이 강함. "계속 기능을 붙인다"는 특성상, 배포 후 새로고침으로 끝나는 웹이 매번 빌드·재설치하는 네이티브보다 압도적으로 빠르다. 판매가 아니라 스토어 발견성·심사도 불필요. "앱처럼 보이는 것"(홈 아이콘·전체화면·카메라·오프라인·푸시)은 PWA가 다 제공.
- **기각:** 네이티브(빌드/재설치 반복, 본인용엔 과함), Fud AI 등 기존 오픈소스 포크(iOS+안드로이드 네이티브 2종 코드베이스라 웹 전환 시 재사용률 0).

### D2. 안드로이드만 대응
- **결정:** iOS 미지원.
- **이유:** 본인이 안드로이드 사용. iOS 웹푸시는 홈 화면 설치 상태에서만 동작하는 등 제약이 많은데, 그걸 신경 쓸 이유가 없음.
- **기각:** 크로스 플랫폼 대응(불필요한 복잡도).

### D3. 데이터는 Neon(Postgres)에 저장. 로컬 저장 안 씀
- **결정:** 모든 기록을 Neon Postgres에.
- **이유:** 브라우저 로컬 저장(IndexedDB)은 그 브라우저 안에 갇히고 캐시 삭제 시 소실됨. 폰·PC 양쪽에서 같은 데이터를 보려면 서버 DB가 필요. 이미 Neon에 익숙함.
- **기각:** IndexedDB/localStorage (동기화·백업 불가).

### D4. ORM은 drizzle-orm
- **결정:** drizzle-orm + Neon serverless driver.
- **이유:** 기존에 쓰던 스택. 타입 안전, 마이그레이션 깔끔, Vercel 서버리스 환경과 궁합 좋음.

### D5. 사진 분석은 Gemini 2.0 Flash
- **결정:** 멀티모달 LLM으로 Gemini 2.0 Flash 사용.
- **이유:** 음식 인식·칼로리 추정에 싸고 빠르고 충분. 구조화 JSON 출력 잘함. 이미 익숙.
- **참고:** "AI"는 별도 학습 모델이 아니라 그냥 사진+프롬프트를 LLM에 던지는 것. 모델 교체는 어댑터 한 군데만 바꾸면 되도록 추상화할 것 (`06` 참고).
- **기각:** 자체 비전 모델 학습(과함), CLIP+DB 매칭(MVP엔 불필요).

### D6. 배포는 Vercel
- **결정:** Vercel.
- **이유:** Next.js 궁합 최상, Vercel Cron으로 푸시 스케줄링까지 한 곳에서 해결.
- **참고:** Netlify도 Scheduled Functions로 가능은 하지만, Cron+서버리스 일관성 때문에 Vercel로 통일.

### D7. 인증은 "최소 보호" — 회원가입 없음
- **결정:** 단일 사용자용. 풀 인증 시스템 없이, 공개 URL이 그냥 노출되지 않도록 가벼운 보호만 건다.
- **이유:** 본인만 씀. 로그인 UX는 마찰만 추가.
- **방법:** `02-architecture.md`의 인증 섹션 참고 (단일 패스코드/쿠키 또는 환경변수 기반 미들웨어).

### D8. 차트는 Recharts
- **결정:** Recharts.
- **이유:** 기존 스택. 일/주/월 막대·라인 차트에 충분.

### D9. 푸시는 web-push + VAPID + Vercel Cron, 스마트 스킵
- **결정:** 표준 Web Push(VAPID 키) + `web-push` 라이브러리, 발송 스케줄은 Vercel Cron. 발송 전 "오늘 이미 기록했나" 확인해 기록 있으면 건너뜀.
- **이유:** 외부 푸시 서비스 없이 자체 완결. 잔소리 안 하는 알림.
- **순서:** 핵심 앱 완성 후 Phase 4에서 얹음 (곁가지 많아 처음부터 넣으면 디버깅 지점 증가).

### D10. 프로젝트명: 밥로그 (BabLog)
- **결정:** 한글 표기 "밥로그", 식별자/도메인 `bablog`.
- **이유:** 밥 + log. 발음 쉽고 뜻 명확. "오늘 밥로그 찍었어?"가 자연스러움.

### D14. 비주얼 톤: 골격은 관습대로, 스킨만 "귀엽게"
- **결정:** 레이아웃·인터랙션(하단 탭 + 가운데 `+`, 끼니 분류, 차트 등 학습된 패턴)은 시중 식사기록 앱 관습을 그대로 따르고, **비주얼 톤만 귀엽고 말랑하게** 간다. 둥근 모서리, 부드러운 색, 음식/마스코트 이모지·일러스트 포인트, 저장 시 가벼운 애니메이션.
- **이유:** 익숙한 UX 관습을 깨면 학습 비용이 들어 불편해짐. 차별화는 구조가 아니라 톤에서 낸다.
- **가드레일:** 장식 폰트·파스텔은 헤더/빈 화면/마스코트 등 포인트에만. **본문 가독성과 버튼 대비는 절대 희생하지 않는다.**
- **확정 디자인 시스템 (2026-06-23, 기록 홈 적용):**
  - 팔레트: 배경 크림 `#FFF8F0`, 카드 `#FFFDFB`, 먹색 텍스트 `#3A332E`, 포인트 코랄 `#FF8A6B`(+ soft `#FFE9E1`), 보조 말차 `#7FB86C`, 회색 `#B8AFA6`, 라인 `#F1E8DE`. (토큰은 `app/globals.css` `@theme`)
  - 타이포: 헤더·숫자 **Jua**(`font-display`), 본문 **Noto Sans KR**(`font-body`). 장식 폰트는 포인트에만, 본문 가독성 우선.
  - 시그니처: **🍚 밥공기 마스코트**(bob 애니메이션) + 오늘 끼니를 🍚 줄로 표시 + 저장 시 pop 토스트.
  - 모션: `pop`/`bob` 키프레임, `prefers-reduced-motion` 존중.
- **인아웃 참고 원칙:** 캐릭터의 "귀여운 온기"는 취하되, 인아웃의 게이미피케이션(점수·배틀)·커뮤니티·공간꾸미기는 **밥로그 비목표라 제외**.
- **후속:** 전용 일러스트 마스코트는 Phase 1.5 브랜딩 패스에서(지금은 이모지로 대체).

### D15. 목표 설정: kcal 목표 대신 "체중 목표"부터, Phase 2에서 검토
- **결정:** 백로그의 "일일 kcal/매크로 목표 + 대비 표시"(`09` Phase 5+)는 그대로 백로그에 두고 당기지 않는다. 대신 **체중 목표**(예: "목표 65kg")를 Phase 2(체중) 작업 때 가볍게 얹을지 검토한다.
- **이유:** 딱딱한 kcal 목표/초과 경고는 "정밀 영양 측정 안 함, 칼로리 강박 대신 가벼운 AI 조언"이라는 포지션(`00` 비목표, D5)과 충돌. 게다가 AI 추정치가 부정확한데 목표 대비를 정밀히 보여주면 신뢰가 깨짐.
- **순서:** kcal 목표는 AI 추정 정확도가 검증된 Phase 3 이후에 백로그에서 재검토.

### D16. 단일 사용자 → 멀티유저로 전환 (D7 일부 대체)
- **결정:** 완전 1인용을 접고, **본인 + 초대한 지인 몇 명**이 각자 자기 데이터로 쓰는 멀티유저로 간다. "유료 앱 대신 무료로 이거 써" 용도. **판매·수익화는 여전히 안 함.**
- **영향:**
  - **스키마** — `users` 테이블 신설 + `meals`·`weight_logs`에 `user_id` FK 추가, 모든 쿼리를 `user_id`로 스코프.
  - **인증** — D7의 "단일 패스코드"를 **사용자별 로그인**으로 교체.
  - **격리** — 푸시 구독(Phase 4)·보고서 등도 사용자별로 분리.
- **범위 가드:** 멀티유저지만 **각자 독립 개인 공간**. 공유·팀·역할·결제는 없음.
- **로그인 방식: 카카오 소셜 로그인.** 지인 대부분 한국 사용자라 카카오가 마찰 최소(구글보다 익숙). Auth.js의 Kakao OAuth provider + Drizzle 어댑터로 붙임(D4 스택 궁합). 준비물: 카카오 개발자 앱 등록 → REST API 키 + Client Secret + 리다이렉트 URI 등록. 되돌리기 비싼 `user_id` 컬럼은 **데이터 쌓이기 전 지금** 넣는다.

### D17. 사진 보관함: Cloudflare R2 (Vercel Blob 대신)
- **결정:** 끼니 사진을 **영구 보관**하고, 저장소는 **Cloudflare R2**(S3 호환). `meals.photo_url`에 공개 URL 저장. 업로드는 서버(`/api/*`)에서만.
- **이유:** 사진은 음식 일기의 핵심 가치(조회의 즐거움·회상). R2는 **egress 비용 0**이라 PWA에서 이미지 반복 서빙에 유리하고 무료 한도가 큼 — `03`이 적어둔 Vercel Blob보다 비용 면에서 나음.
- **구현:** `@aws-sdk/client-s3`로 R2에 PUT, 공개 버킷(r2.dev 또는 커스텀 도메인)에서 서빙. 환경변수 `R2_*`. (MVP는 저장 시 업로드, 재촬영 시 고아 객체는 일단 허용.)
- **기각:** 분석 후 사진 폐기(히스토리가 밋밋), Vercel Blob(전송 과금).

### D18. 기록 입력 개선: 끼니 분류 + 항목별 양 보정 + 수동 입력
- **끼니 분류:** `meals.meal_type`(아침/점심/저녁/간식) 추가. 먹은 시간으로 자동 추정 + 보정 화면에서 탭 변경. (기존 행 호환 위해 nullable.)
- **보정 모델 변경:** 합계 직접 수정 → **항목별 양(숫자) + 단위(g/개/ml/인분/그릇 등)**, 양을 바꾸면 매크로가 비례 스케일. 합계는 항목 합으로 파생. 빠른 조정용 ½·1·1.5·2 배수 버튼은 보조. (AI가 항목마다 양+단위+기준 매크로를 반환하도록 출력 스키마 확장.)
- **수동 입력(사진 없이): DB 우선 + AI 폴백.** 음식 검색→선택→양(g/인분) 입력→영양성분. DB에 없으면 텍스트→AI 추정으로 폴백. 같은 보정 화면 재사용.
  - **음식 DB:** 식약처 식품영양성분 공공 API(data.go.kr, 한식 커버). API 키는 사용자 발급 필요. (Open Food Facts는 포장식품 위주·한식 약해서 보류.)
  - **구현 메모(식약처 API):**
    - 엔드포인트: `https://apis.data.go.kr/1471000/FoodNtrCpntDbInfo02/getFoodNtrCpntDbInq02`
    - 키: env `FOOD_API_KEY`(data.go.kr 일반 인증키). `serviceKey`로 전달, `type=json`(기본 xml). 일일 트래픽 10000.
    - 요청 파라미터: `pageNo`, `numOfRows`, `type`, `FOOD_NM_KR`(식품명 검색), `FOOD_CAT1_NM`(대분류), `MAKER_NM`, `DB_CLASS_NM` 등.
    - **[해결 2026-06-23] 검색·필드 확인 완료.** 이전 "totalCount 0"은 **검색어 한글을 URL 인코딩 안 한 아티팩트**였음(Git Bash raw 한글). 제대로 인코딩하면 부분 매칭으로 정상 동작: 김치찌개 356·된장찌개 364·비빔밥 718·신라면 51건 등, 전체 30만 건. **응답 필드 매핑 확정:** `AMT_NUM1`=에너지(kcal)·`AMT_NUM3`=단백질·`AMT_NUM4`=지방·`AMT_NUM6`=탄수화물 (모두 `SERVING_SIZE` 기준, 보통 100g/100mL)·`Z10500`=1회 제공 중량(기본 양으로 사용)·`MAKER_NM`=제조사. 검증 코드는 `lib/food/foodApi.ts`.
    - **[결정 2026-06-23] 방식 A(라이브 API) 확정.** (B) 자체 import를 권했던 근거가 "0건"이었는데 그게 오인이었고, 라이브 커버리지·부분매칭이 충분함. 소규모 사용이라 일 10,000 호출 한도 여유, **infra 0**(foods 테이블·import job 불필요)이라 A가 압도적으로 단순. 매 검색마다 식약처 호출(`/api/foods/search`), DB에 없으면 **AI 텍스트 추정 폴백**(`/api/foods/estimate`). (B)는 검색 품질/지연이 문제될 만큼 트래픽이 커지면 그때 재검토.
- **API 호출 절약(Gemini 무료 일 500회):** **분석은 끼니당 1회**가 원칙 — 항목별 호출 금지, 보정은 전부 클라이언트(0회). DB 검색은 호출 0회. 순수 수동·즐겨찾기로 추가 절약.

### D19. 음식 DB(`foods`) 도입 확정 — 캐싱 시점 + 즐겨찾기/직접등록 UX
- **결정:** 백로그(2026-06-24)로 미뤄두던 음식 DB를 **Phase 6**으로 착수(`09`). 세부 결정 3가지:
  1. **저장 범위·시점:** 처음엔 "AI 추정만 저장"으로 좁게 잡았다가, 그러면 식약처 DB에 잘 매칭되는 흔한 한식(가장 자주 반복되는 음식)이 즐겨찾기에 영영 못 뜨는 문제를 구현 전 점검 중 발견(2026-06-28) → **수동 검색으로 추가한 음식은 출처(식약처/AI/직접등록) 안 가리고 저장**으로 확장. 추정/검색 직후가 아니라 **끼니를 실제로 저장하는 시점**에, 사용자가 보정 화면에서 고친 최종값으로 `foods`에 upsert. "AI 재호출 회피"는 그중 AI 추정 항목에서만 의미 있는 부수 효과. 사진 분석(비전) 항목은 범위 밖.
  2. **사용횟수는 사용자별로 분리(같은 날 추가 발견, 2026-06-28):** `foods.usage_count`를 전역 컬럼으로 두려 했으나, 멀티유저(D16)에서 다른 사용자의 사용량이 내 "자주 먹는 음식"에 섞이는 문제 발견 → 즐겨찾기 테이블을 `user_favorite_foods`에서 **`user_foods`(사용자별 `usage_count`+`is_favorite`)로 확장**. `foods`는 영양정보(객관적 사실)만 전역, 사용빈도/즐겨찾기는 사용자별(`03` 참고).
  3. **즐겨찾기 노출:** 검색창이 비어있는 초기 상태에 이 사용자의 즐겨찾기 + `usage_count` 내림차순 "자주 먹는 음식"을 단일 리스트로 노출(MyFitnessPal/Lose It 등 식단 앱 업계 표준 — Frequent/Recent 패턴). 탭 구조 대신 단일 리스트로 단순화(D14 마찰 최소화). 항목마다 별 아이콘으로 즐겨찾기 토글 — 아직 `foods`에 없는 식약처 결과를 즐겨찾기하면 그 시점에 upsert.
  4. **직접 등록 진입점:** 검색 결과 없을 때 AI 추정 버튼 옆에 "직접 등록" 버튼("Can't find it? Create a food" 표준 패턴) — 별도 메뉴로 분리 안 함.
  5. **(1차 실사용 피드백, 2026-06-28)** kcal·단백질·탄수·지방 4개 필드 전부 필수화 — `foods`가 전역 공유라 빈 칸을 0으로 묵인하면 다른 사용자에게도 잘못된 영양정보가 퍼짐. 직접 등록한 음식은 **삭제만 지원, 수정은 없음** — 본인(`created_by` 일치)이 등록한 `source: user` 행만 지울 수 있고, 잘못 입력했으면 지우고 다시 등록(이미 저장된 끼니는 스냅샷이라 영향 없음). `gov`/`ai` 출처는 "내 것"이 아니라 삭제 UI 자체가 없음.
- **이유:** 03/06에 이미 설계돼 있던 백로그를 구체 Phase로 당기면서, 실제 UX/구현 결정을 확정. `foods` 스키마도 `FoodHit`과 같은 `basis_amount`+`unit` 수치 패턴으로 맞춤(보정 화면에서 양 스케일링 가능하도록, 기존 `serving_desc` 텍스트안은 폐기).
- **참고:** `03`(`foods`/`user_foods` 확정 표시) · `06`(조회 우선순위 확정) · `09`(Phase 6 신설).

- **사진 AI 인식 vs 음식 DB 정확도 하이브리드.** AI는 인식·양 추정에 강하고 DB(식약처)는 영양수치가 더 정확 — 인식된 음식명으로 DB 재조회 후 매칭되면 교체하는 2단계 파이프라인 제안(`06` 참고). 적용 시점 미정.
