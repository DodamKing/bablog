# 06 · AI 파이프라인 (AI Pipeline)

> 핵심 통찰: "AI"는 특별한 학습 모델이 아니라 **멀티모달 LLM에 사진+프롬프트를 던지고 구조화 JSON을 받는 것**.
> 모델은 `lib/ai/` 어댑터 뒤에 격리 → 교체 한 군데에서.

## A. 사진 → 영양 추정

### 호출
- 모델: Gemini 2.0 Flash.
- 입력: 사진(inline base64) + 프롬프트.
- 출력: **JSON만**. 프롬프트에서 마크다운/설명 없이 JSON만 내라고 강하게 못박고, 응답을 안전 파싱(코드펜스 제거 후 `JSON.parse`).

### 프롬프트 설계 (의도)
시스템/프롬프트에 담을 핵심 지시:
- 너는 음식 사진에서 개별 음식 항목과 양을 추정하는 전문가다.
- 셀 수 없는 항목은 그램으로 추정하고, 한국 음식(한식)에 특히 강하게 인식하라.
- 각 항목과 전체에 대해 kcal/단백질/탄수/지방을 추정하라.
- 불확실하면 보수적으로 범위 중간값을 쓰고, 추정임을 전제하라.
- **JSON 외 아무것도 출력하지 마라.**

### 출력 JSON 스키마
```json
{
  "items": [
    { "name": "김치찌개", "qty": "1 그릇(약 400g)", "kcal": 240, "protein_g": 16, "carb_g": 12, "fat_g": 13 }
  ],
  "total": { "kcal": 650, "protein_g": 28, "carb_g": 70, "fat_g": 22 },
  "confidence": "low | medium | high",
  "notes": "사진만으로 추정한 근사값"
}
```
- 클라이언트는 `total`을 기본 표시하되 모두 **수정 가능**하게.
- 원본 응답은 `meals.ai_raw`에 보관(보정 전, 재분석/디버깅).

### 정확도 방어
- AI 추정은 부정확 → **저장 전 보정 화면이 1차 방어선**.
- (선택, 후순위) 바코드/음식 DB로 가공식품 보정.

### 조회 우선순위 (Phase 6, 확정) — 반복 AI 호출 줄이기
같은 음식을 매번 AI에게 새로 추정시키는 게 비효율(2026-06-24 사용자 지적) → 3단계 조회. **"캐시"가 아니라 전역 공유 영구 DB**(`foods`, `03` 참고) — 한 번 추정/등록되면 모든 사용자가 재사용:
1. 식약처 DB 라이브 검색 (D18, 가장 정확)
2. `foods` 테이블 — 과거에 끼니로 저장된 적 있는 음식(식약처 매칭·AI 추정·직접등록 가리지 않음, 영양정보는 전역 공유). `/api/foods/search`가 식약처 결과와 병합해 반환.
3. 그래도 없으면 AI 텍스트 추정
- **저장 시점은 검색이 아니라 끼니 저장 시점.** 결과를 보정해서 "저장"을 누르는 순간, 출처와 상관없이(식약처 매칭이든 AI 추정이든) 보정된 최종값으로 `foods`에 upsert(이름 매칭, 전역 영양정보는 기존 있으면 안 바꿈) + 이 사용자의 `user_foods`에서 `usage_count` +1 — 다음부턴 2번에서 바로 잡힘. AI 호출 회피는 이중 효과 중 AI 추정 항목에서만 의미 있을 뿐(D19 보강, 2026-06-28).

**즐겨찾기/직접등록 UX(D19, 식단 앱 업계 표준 — MyFitnessPal/Lose It류 Frequent·Recent 패턴 참고):**
- 검색창이 비어있는 초기 상태(검색화면 진입 직후)에 **이 사용자의** 즐겨찾기 우선 + `usage_count` 내림차순 "자주 먹는 음식"을 **단일 리스트**로 노출(탭 구조 대신 — D14 마찰 최소화 기조). `user_foods` 기준이라 다른 사용자 사용량과 안 섞임(`03` 참고). 타이핑 없이 한 탭으로 재입력.
- 식약처/`foods` 결과 항목마다 별(☆/★) 아이콘으로 즐겨찾기 토글(`user_foods.is_favorite`).
- 검색 결과가 없을 때 AI 추정 버튼 옆에 "직접 등록"(이름+기준량+kcal/매크로) 버튼 — "Can't find it? Create a food" 표준 패턴, 별도 메뉴로 분리하지 않고 검색 흐름 안에서 바로 이어짐.

### 사진 인식 + DB 영양정보 하이브리드 (제안, 백로그)
AI 비전은 "사진에 뭐가 있고 대략 얼마나 되는지"(인식 + 양 추정)에 강하고, 영양 수치 자체는 식약처 DB가 더 정확함.
→ 사진 분석 결과의 `items[].name`으로 식약처 DB를 재조회해, 매칭되면 그 음식의 kcal/매크로(+나트륨/당류 등)로 **교체**하고, 매칭 안 되면 AI 추정치를 그대로 둔다. 양(g)은 AI의 비전 추정을 신뢰.

## B. 보고서 / 조언 (Phase 3, 구현됨)

### 호출
- 입력: 기간 집계(일별 총 kcal·매크로, 끼니 수, 체중 추세 포인트) + **최근 추이(선택, 비교용)**를 텍스트로 요약해 프롬프트에 첨부.
- 출력: 자연어 보고서 + 조언. (구조 필요하면 섹션 JSON으로 받아 렌더해도 됨.)
- **최근 추이(2026-06-24 추가):** "AI가 사용자를 학습"하는 게 아니다(Gemini 호출은 매번 독립적, 모델이 누적 학습하지 않음) — 대신 매 호출마다 `meals`에서 **이번 기간 이전 N개 기간(기본 4개, `TREND_BUCKETS`)을 추가로 조회**해 기간별 평균을 프롬프트에 날짜range와 함께 끼워준다(`app/api/report/route.ts`의 `loadTrend`). 그러면 "지난주보다 늘었다/줄었다" 또는 여러 주의 흐름까지 자연스러운 비교로 나옴. `meals`가 이미 영구 보관되니 새 저장소 없이 조회 범위만 넓히면 됨 — 데이터가 없는 과거 버킷은 건너뜀.
- **저장/캐싱(2026-06-24 결정):** AI 호출 자체는 캐싱 안 함(누를 때마다 새로 생성, `09`)이지만, **생성된 텍스트는 기간(주/월)당 최신 1개씩 `reports`에 저장**해서 탭 이동해도 안 사라지게 함(`03` 참고). 통계 카드는 항상 라이브 재계산, 저장된 건 AI 텍스트뿐. 조회 시(`GET /api/report`) 저장 당시 집계와 지금 집계를 `summaryFingerprint`로 비교해 `stale`(변경 여부)만 알려주고, **자동으로 다시 호출하지는 않음** — 사용자가 "다시 생성"을 눌러야 새 AI 호출이 일어난다(예측 가능한 비용 유지).

### 프롬프트 가이드 (중요 — 톤)
- 데이터에 근거해 **사실 중심 요약**: 평균 섭취, 매크로 분포, 패턴, 체중과의 관계.
- 조언은 **균형 잡히고 비강압적**으로. 극단적 칼로리 제한·단식·특정 음식 금지 같은 강한 처방을 하지 말 것. "참고용 관찰"의 톤.
- 의학적 단정·진단 금지. 필요 시 "전문가 상담" 정도만.
- 사용자가 1인(본인)이라 개인화는 자유롭게.
