## Phase 6 · 음식 DB (즐겨찾기 + 직접 등록 + AI 캐싱)
**읽을 문서:** `03`, `06`
> 같은 음식을 매번 AI에게 새로 추정시키는 비효율을 없애고, 자주 먹는 음식(식약처 매칭이든 AI 추정이든)은 검색 없이 한 탭으로 다시 기록할 수 있게 한다. "캐싱"은 별도 동작이 아니라 저장 흐름에 자연히 포함됨 — 보정해서 끼니를 저장하면 그 최종값이 출처를 안 가리고 `foods`에 같이 기록됨(D19).
- [x] 스키마: `foods`(전역 영양정보, `source`: gov\|ai\|user, `basis_amount`+`unit`, `name` unique, `created_by`) + `user_foods`(사용자별 `usage_count`+`is_favorite`, PK `user_id`+`food_id`) 테이블 추가, `db:push` 마이그레이션
- [x] `/api/foods/search`: 식약처 라이브 검색 + `foods`(+`user_foods` 조인) 조회를 이름으로 병합(중복 제거)해 반환. 검색어 없이 호출(검색화면 진입 직후) 시 이 사용자의 즐겨찾기 우선 + `usage_count` 내림차순으로 "자주 먹는 음식" 목록 반환
- [x] 즐겨찾기 토글 엔드포인트 — `foodId`(이미 `foods`에 있는 경우) 또는 식약처 `FoodHit` 전체(아직 없는 경우, upsert 후 `user_foods.is_favorite` 토글)를 받아 처리
- [x] 직접 등록: 검색 결과 없을 때 AI 추정 버튼 옆 "직접 등록" 버튼 → 이름+양/단위+kcal/매크로 입력 폼(4개 필드 전부 필수, 클라이언트+서버 검증) → `foods`에 `source: user`로 저장 + 보정 화면에 바로 추가
- [x] `/api/meals` POST: 저장되는 항목 중 수동 검색으로 추가된 것(식약처/foods/AI/직접등록 가리지 않음, 사진 분석 항목은 제외)은 `foods`에 upsert(이름 매칭, 기존 있으면 영양정보 안 바꿈) + 이 사용자의 `user_foods.usage_count` +1
- [x] 검색 화면 UI(`app/(app)/page.tsx`): 검색창 비어있을 때 즐겨찾기/자주 먹는 음식 리스트, 모든 음식 항목에 별(☆/★) 아이콘
- [x] 직접 등록한 음식 삭제 — `/api/foods` DELETE(본인이 등록한 `source: user` 행만, `created_by` 일치) + 자주먹는 목록에 🗑 버튼(`ConfirmDialog` 재사용). 수정은 없음(지우고 다시 등록하는 방식으로 대체) — 이미 저장된 끼니는 스냅샷이라 영향 없음
- [x] **(1차 실사용 피드백, 2026-06-28)** 직접등록 폼이 검색어를 지우면 사라지는 버그 수정, 매크로 필드 빈칸이 0으로 묵인되던 문제 수정(4개 필드 필수화)
- [x] **(사용자)** 폰에서 재확인 — 검색·즐겨찾기·직접등록·삭제·AI 재호출 안 되는지(2026-06-29 통과)
- **완료 기준:** 식약처에 매칭되는 흔한 음식도, AI로 추정한 음식도 똑같이 즐겨찾기/자주먹는 목록에서 한 탭으로 다시 기록할 수 있고, 한 번 저장된 AI 추정 음식은 같은 이름으로 또 AI를 부르지 않는다.

## Phase 7 · 목표 칼로리/매크로 시스템
**읽을 문서:** `01`(D15·D20), `03`(`user_profiles`·"계산값"), `05`(1번 화면·체중 절)

- [x] 스키마: `user_profiles`(`user_id` pk, `birth_year`, `height_cm`, `gender`, `activity_level`, `goal_type`, `target_weight_kg`, `weekly_rate_kg`) 추가, `db:push`
